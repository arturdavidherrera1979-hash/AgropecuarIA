AGROPECUARIA – Guía del repositorio y estructura

Documento para alinear al equipo: acceso al repo, reglas de uso y propósito de cada carpeta.

Actualizado: 2025-12-26

0) Acceso al repositorio (para nuevos colaboradores)

Link oficial del repositorio: https://github.com/arturdavidherrera1979-hash/dronia

Paso previo: aceptar la invitación a colaborar en GitHub (si te llegó por mail o notificación). Sin aceptar la invitación, un repo privado puede verse como “no existe” o “no encontrado”.

¿Necesito instalar algo?

  • Solo para leer / revisar: no. Con navegador alcanza (GitHub Web).
  • Para editar documentos simples: podés hacerlo desde la web (botón Edit / crear PR).
  • Para trabajar con código: recomendado instalar GitHub Desktop o Git (línea de comandos) para clonar y versionar en tu PC.

Opción A: GitHub Web (sin instalaciones)

  • Entrá al link del repo.
  • Podés navegar carpetas, ver Issues, revisar PRs.
  • Para cambios chicos en docs: editar archivo → “Commit changes” → idealmente vía Pull Request.

Opción B: GitHub Desktop (recomendado para no técnicos)

  • Instalá GitHub Desktop.
  • Iniciá sesión con tu cuenta GitHub.
  • File → Clone repository → elegí dronia → seleccionar carpeta local.
  • Editar en tu editor favorito → commit → push → PR desde Desktop.

Opción C: Git (CLI) para desarrolladores

  • Instalá Git for Windows.
  • Cloná por HTTPS y autenticá con GitHub (token/credenciales). Con GitHub CLI, suele bastar con: gh auth setup-git.

1) ¿Para qué sirve este repositorio?

El repositorio AGROPECUARIA es el “centro de gravedad” del proyecto: acá viven el código, la documentación y los artefactos de gestión (issues, plantillas, minutas). La idea es que cualquier participante pueda entrar, entender el estado y contribuir sin depender de memoria oral.

Regla simple

  • Todo lo que se decide (técnico/comercial/operativo) debe quedar reflejado en docs/ y/o en un Issue.
  • Todo lo que se ejecuta (código/pipeline) vive en src/ y se prueba desde tests/.
  • Todo lo que se repite (plantillas, SOPs) debe quedar en el repo para que sea reutilizable.

3) ¿Para qué sirve cada carpeta?

Ruta Propósito ¿Quién la usa más? Contenido típico
.github/ Automatización y estándares en GitHub (CI, plantillas, PRs). Todo el equipo Workflows de Actions, templates de Issues/PRs.
.github/workflows/ Pipeline de CI (integración continua) que corre automáticamente en pushes/PRs. Dev / QA ci.yml para ejecutar tests y validar que el repo esté sano.
.github/ISSUE_TEMPLATE/ Plantillas para crear issues con formato consistente. PM / Equipo Formularios de tareas: negocio, research técnico, minutas.
docs/ Documentación viva: visión, roadmap, arquitectura, definiciones. Todo el equipo Visión, roadmap, data model, MVP definition, decision log.
business/ Material del modelo de negocio (Canvas, hipótesis, costos/beneficios). Negocio Business Model Canvas, propuesta de valor, segmentación, unit economics.
commercial/ Go-to-market, pricing, mensajes, canales y proceso de venta. Comercial GTM, pricing, scripts de outreach, objeciones/respuestas.
hardware/ Investigación y decisiones de drones/sensores (stack operativo). Técnico/Operaciones Comparativas de plataformas, sensores, criterios, costos.
meetings/ Minutas y plantillas de reuniones para no perder decisiones. Todo el equipo Template de minuta y minutas por fecha.
src/ Código fuente del proyecto (la “parte ejecutable”). Dev Módulos Python, pipeline, herramientas internas.
src/dronia_core/ Núcleo del código: entrypoint y lógica principal del servicio/pipeline. Dev main.py, librerías internas, utilidades comunes.
tests/ Pruebas automatizadas para asegurar que cambios no rompan lo existente. Dev / QA Tests unitarios/integración (ej. pytest).

4) Archivos importantes (qué son y por qué importan)

Archivo Para qué sirve
README.md Portada del repo: qué es AGROPECUARIA, cómo arrancar y dónde mirar primero.
requirements.txt Dependencias Python mínimas para correr el proyecto y los tests.
Dockerfile Contenedor para ejecutar el proyecto de forma consistente (útil para despliegues/pipeline).
.gitignore Evita subir basura al repo (entornos virtuales, caches, logs).
LICENSE Define los términos de uso del código (importante para colaboradores y futuros partners).
.github/PULL_REQUEST_TEMPLATE.md Checklist para PRs (calidad mínima y contexto).
.github/workflows/ci.yml CI: ejecuta tests en cada push/PR. Si se pone rojo, es una alarma útil.

5) Cómo contribuir sin romper nada

  • Documentación: edita/crea archivos en docs/, business/, commercial/, hardware/ y vincula el cambio a un Issue.
  • Código: trabaja en ramas (branch) y abre PRs. Asegurate de que CI quede verde.
  • Minutas: cada reunión deja una minuta en meetings/ y actualiza decisiones relevantes en docs/.

Tip: si un cambio es “decisión” (no solo texto), agregá una sección Decision log en el doc correspondiente.
